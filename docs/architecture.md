# IAM dApp Architecture Notes

## Actors
- **Holder**: owns DID, receives credentials, grants/revokes verifier consent.
- **Issuer**: issues and revokes credential status anchors.
- **Verifier**: verifies credential integrity/status/consent and logs verification events.
- **Admin**: authorizes trusted issuers in `AccessControl`.

## On-Chain Contracts
- `DIDRegistry.sol`
  - Registers DIDs and controller metadata.
  - Supports key/service updates and deactivation events.
- `CredentialStatus.sol`
  - Anchors credential hash + off-chain pointer (`ipfsCID` field used as generic CID).
  - Maintains bitmap revocation status.
  - Emits verification log events for auditability.
- `AccessControl.sol`
  - Tracks stakeholder roles and DID associations.
  - Enforces holder consent for verifier access checks.
  - Integrates revocation status in `hasConsent`.

## Off-Chain Credential Storage
- Implemented via encrypted local credential DB (`scripts/offchainStore.js`) to satisfy the off-chain storage requirement.
- Flow:
  1. Credential JSON is canonicalized.
  2. Payload is encrypted with AES-256-GCM.
  3. Record is persisted under `offchain-db/credentials.json`.
  4. A deterministic CID-like identifier is derived from encrypted payload hash.
  5. On verify, record is decrypted and re-hashed for integrity comparison.

This storage adapter can be swapped with IPFS while preserving the same contract interactions.

## Demo Interface
- A lightweight browser dashboard is available under `web/`.
- It supports wallet connection, contract binding, DID registration, role/consent actions, credential issuance, integrity checks, and revocation flows.
- The dashboard loads contract addresses from `deployments.json` generated by `scripts/deploy.js`.

## Identity Lifecycle Mapping
1. **Registration**
   - Holder/Issuer/Verifier each register DID in `DIDRegistry`.
   - Holder and Verifier self-register roles in `AccessControl`.
   - Admin authorizes Issuer role.
2. **Issuance**
   - Issuer writes credential hash + CID anchor to `CredentialStatus`.
3. **Presentation/Verification**
   - Holder grants verifier consent for a credential.
   - Verifier fetches payload off-chain, recomputes hash, checks:
     - `verifyCredentialIntegrity`
     - `isRevoked`
     - `hasConsent`
   - Verifier writes `VerificationLogged` event.
4. **Revocation**
   - Holder may revoke consent.
   - Issuer may revoke credential status via bitmap.

## Trust And Privacy Notes
- No raw credential payload is stored on-chain.
- DIDs and hashes remain publicly linkable metadata; report should discuss this tradeoff.
- Production hardening should replace dev secret handling with managed key material and formal encryption policy.
